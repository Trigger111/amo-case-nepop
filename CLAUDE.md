# AMO Case — Продуктовый аналитик (Genesis Analytics Camp 3.0)

Первый продуктовый кейс. Приложение по подписке (персональные тренировки), продаётся
через квиз-воронку на сайте, рынки US/EU/Tier-1, бесплатной версии нет. ~6 месяцев назад
цену подняли 6.99 → 9.99/нед на основе A/B-теста, который показал, что конверсия не упала.
Сейчас CEO сомневается: цена +42%, а в Revenue такого роста не видно. Разбираемся.

## КАК СО МНОЙ РАБОТАТЬ (важнее всего)

Пользователь (Александр) делает кейс, чтобы **научиться** — он пишет **весь код анализа сам**.
- Новичок в Python, SQL (проседает на джоинах), A/B-статистике.
- Моя роль: **критический наставник**. Объясняю концепты, ревьюю его код, ловлю ошибки в
  рассуждениях, оспариваю необоснованные предположения.
- **НЕ писать за него код анализа и не выдавать готовые посчитанные ответы.** Даю по одному
  строительному блоку / одному новому механику + цель — он собирает сам.
- Маленькие шаги. Не грузить плотными однострочниками (переваривает тяжело) — раскладывать.
- Проверять данные молча в своём терминале можно, но не раскрывать числа, которые он должен
  вывести сам.
- Язык общения: русский.

- Переменные в ноутбуке: `df_test_data`, `df_order_data`, `df_otp_data`, `df_refunds_data`,
  `df_user_data`, `df_marketing_spend_data`, и очищенный `df_test_clean`.

## Данные (6 таблиц)

- **test_data** (события A/B-воронки, янв–фев): `user_id`, `event_date`, `event_name`
  (click / purchase / **otp_purchase** — апсел; в презентации он назван sub_purchase),
  `group` (control=6.99 / test=9.99), `country_group` (EU/USA/Tier1), `platform`,
  `payment_method`, `product_name`, `channel`. 1 строка = 1 событие.
- **order_data** (ребилы, раскатка на 50%, янв–мар): `user_uuid`, `date_started`,
  `Product Name`, `order_date`, `Trial Period Days`(7), `Rebill Period Days`(7). Много строк/юзер.
- **otp_data**: `user_uuid`, `Product Name`, `revenue`(=15). Покупатели OTP-апсела.
- **refunds_cb_data**: `user_uuid`, `Product Name`, `refund`(0/1), `chargeback`(0/1), `price`.
  Чарджбек = возврат стоимости + штраф $30.
- **user_data**: `user_uuid`, `region`, `platform`, `payment_type`, `age_segment`. 1 строка/юзер.
- **marketing_spend_data**: `date`, `channel`, `country_group`, `group`, `clicks`, `cpc`, `spend`.

Грабли: ключ джоина — `user_id` в test_data, но `user_uuid` в остальных. Регистр product_name
разный ("Weekly 6.99" vs "weekly 6.99").

## Задания

1. **A/B:** правильно ли решили по конверсии в покупку? Как цена повлияла на конверсию в
   основной продукт И в апсел для разных сегментов?
2. **LTV на 1.5 мес** для 6.99 vs 9.99 по сегментам, с учётом рефандов, чарджбеков, OTP-апселов.
3. Реально ли ARPU вырос на +42% при цене +42%? Сколько заработали/потеряли за **26.01–16.03**
   из-за перехода на дорогую цену? Как изменился доход с 1 юзера?
4. **ROAS** по сегментам (канал, регион, ценовая группа). Меняет ли это рекомендацию по цене?
   Изменилась ли окупаемость при новой цене vs старой?
Финал: рекомендация по цене.

## Прогресс

### Блок 0 — качество данных: ГОТОВО
- 1131 полный дубль строк, все в control/click → почищено в `df_test_clean` (`drop_duplicates()`).
- **Контаминация Tier1 (ключевая находка):** в тестовой группе ВСЕ Tier1-юзеры по цене 6.99,
  а не 9.99 (видно через `crosstab(country_group, product_name)`). То есть Tier1 никогда не
  тестировался валидно → это случайный A/A. Валидное ценовое сравнение = только EU+USA.
  Ещё это значит, что раскатка 9.99 была неравномерной → часть головоломки с Revenue.
- test_data = окно A/B-теста; order/user/refunds/otp = поздняя раскатка на 50% → разные
  популяции и периоды, не смешивать.

### Задание 1 — конверсия: В ПРОЦЕССЕ
Конверсия click→purchase (уник. покупатели / уник. кликеры) по region×group,
посчитана пользователем самостоятельно (вложенный цикл):
| регион | control 6.99 | test 9.99 |
|---|---|---|
| EU | 5.24% | 5.41% |
| USA | 7.03% | 7.11% |
| Tier1 (обе 6.99) | 6.42% | 6.89% |

Наблюдение: конверсия 9.99 ≥ 6.99 везде, но дельты крошечные. Разброс в Tier1-A/A (0.47 п.п.)
БОЛЬШЕ, чем «эффект» в EU+USA (0.13 п.п.) → эффект, вероятно, в пределах шума. Нужен стат-тест.
